# القسم الثالث: بنية النظام Architecture System

## 1. مقدمة عن بنية النظام

تم تصميم المشروع وفق بنية **Service-Oriented Architecture - SOA**، وهي بنية تعتمد على تقسيم النظام إلى مجموعة خدمات مستقلة، بحيث تكون كل خدمة مسؤولة عن جزء واضح من العمل. الهدف من هذا التصميم هو جعل النظام أكثر تنظيماً، وأسهل في التطوير والصيانة، وأقرب إلى طريقة بناء الأنظمة الحقيقية في بيئات العمل.

بدلاً من أن يكون المشروع عبارة عن برنامج واحد يحتوي على كل العمليات، تم فصل الوظائف الأساسية إلى خدمات مثل المعالجة المسبقة، الفهرسة، الاسترجاع، الترتيب، تحسين الاستعلام، والتقييم. هذا الفصل يجعل فهم المشروع أسهل؛ فإذا أردنا مثلاً تعديل طريقة ترتيب النتائج، لا نحتاج إلى تغيير خدمة المعالجة أو خدمة التقييم.

بصراحة، هذا النوع من التنظيم يعطي المشروع طابعاً احترافياً أكثر، لأنه لا يكتفي بتطبيق خوارزميات البحث، بل يضعها داخل بنية قابلة للتشغيل والتوسعة.

## 2. فكرة SOA في المشروع

تعتمد SOA على أن كل جزء من النظام يقدم وظيفة محددة عبر واجهة واضحة. في هذا المشروع، تم تطبيق هذه الفكرة باستخدام **FastAPI** لبناء REST APIs بين الخدمات.

الفكرة العامة هي:

- كل خدمة لها مسؤولية محددة.
- التواصل بين الخدمات يتم عبر HTTP/REST.
- توجد نقطة دخول رئيسية للنظام وهي API Gateway.
- يمكن تشغيل الخدمات معاً من خلال Gateway واحد، أو تشغيل كل خدمة بشكل مستقل على منفذ خاص.

هذا التصميم يساعد على تقليل التشابك بين أجزاء المشروع. فمثلاً، واجهة المستخدم لا تحتاج أن تعرف تفاصيل TF-IDF أو BM25 أو qrels، بل ترسل طلباً إلى Gateway، وهو يتكفل بتوجيه الطلب إلى المكان الصحيح.

## 3. المكونات الرئيسية للنظام

يتكون النظام من المكونات التالية:

| المكون | الوظيفة |
|---|---|
| Frontend | واجهة المستخدم التفاعلية المبنية باستخدام Streamlit |
| API Gateway | نقطة الدخول الموحدة التي تستقبل الطلبات وتوجهها للخدمات |
| Preprocessing Service | معالجة النصوص وتنظيفها قبل استخدامها |
| Indexing Service | بناء الفهارس والنماذج والتحقق من جاهزيتها |
| Retrieval Service | تنفيذ البحث باستخدام طريقة واحدة مثل TF-IDF أو BM25 |
| Ranking Service | تنفيذ استراتيجيات الدمج والترتيب الهجين |
| Query Refinement Service | تحسين الاستعلامات وتقديم الاقتراحات |
| Evaluation Service | تقييم النتائج باستخدام qrels والمقاييس القياسية |
| Shared Layer | ملفات مشتركة مثل schemas، إعدادات البيانات، وإدارة المحرك |
| SQLite Document Store | قاعدة بيانات محلية لحفظ النصوص الأصلية للوثائق |

## 4. الرسم النصي لبنية النظام

يمثل الشكل التالي البنية العامة للنظام:

```text
User
  |
  v
Streamlit Frontend
  |
  v
API Gateway - FastAPI : Port 8000
  |
  +--> Query Refinement Service
  |
  +--> Retrieval Service
  |       |
  |       +--> TF-IDF / BM25 / Index / Word2Vec / BERT
  |
  +--> Ranking Service
  |       |
  |       +--> Serial Fusion / Parallel Fusion / RRF
  |
  +--> Indexing Service
  |       |
  |       +--> Engine Cache / Model Status
  |
  +--> Evaluation Service
          |
          +--> Qrels / Metrics / Reports

Ranking Engine
  |
  +--> Processed Data JSON
  |
  +--> SQLite Original Documents DB
  |
  +--> Cache Files
```

هذا المخطط يوضح أن واجهة المستخدم لا تتعامل مباشرة مع كل تفاصيل النظام، بل تمر عبر API Gateway، ثم يقوم الـ Gateway بتنسيق العمل بين الخدمات.

## 5. API Gateway

يعد **API Gateway** نقطة الدخول الرئيسية للنظام. وظيفته استقبال الطلبات القادمة من واجهة المستخدم ثم تحديد الخدمة المناسبة لتنفيذ الطلب.

يقوم Gateway بالمهام التالية:

- استقبال طلب البحث من الواجهة.
- استدعاء خدمة تحسين الاستعلام.
- اختيار خدمة الاسترجاع أو خدمة الترتيب حسب الطريقة المطلوبة.
- جمع النتائج وإرجاعها بصيغة موحدة.
- توفير endpoints لمعرفة حالة النظام والبيانات.
- تشغيل warmup للمحرك عند بدء العمل لتقليل زمن أول بحث.

الملف الأساسي:

`gateway/app.py`

المنفذ المستخدم في التشغيل الموحد:

`8000`

## 6. الخدمات المستقلة والمنافذ

يمكن تشغيل الخدمات بشكل مستقل، وكل خدمة تعمل على منفذ خاص. هذا مفيد عند الرغبة في اختبار خدمة محددة أو توضيح مفهوم SOA بشكل عملي.

| الخدمة | المنفذ | المسار البرمجي |
|---|---:|---|
| Preprocessing Service | 8001 | `services.preprocessing.app:app` |
| Indexing Service | 8002 | `services.indexing.app:app` |
| Retrieval Service | 8003 | `services.retrieval.app:app` |
| Ranking Service | 8004 | `services.ranking.app:app` |
| Query Refinement Service | 8005 | `services.query_refinement.app:app` |
| Evaluation Service | 8006 | `services.evaluation.app:app` |
| API Gateway | 8000 | `gateway.app:app` |

يمكن تشغيل الخدمات المستقلة باستخدام:

```bash
python run_services.py all
```

أما التشغيل العملي الأبسط أثناء العرض فيتم عبر:

```bash
python run_gateway.py
```

## 7. Shared Layer

توجد طبقة مشتركة بين الخدمات داخل مجلد:

`shared`

هذه الطبقة تحتوي على العناصر المشتركة التي تحتاجها أكثر من خدمة، مثل:

- تعريف الـ datasets.
- نماذج Pydantic المستخدمة في الطلبات والاستجابات.
- إدارة محرك البحث المحمل في الذاكرة.
- التعامل مع قاعدة بيانات الوثائق الأصلية.

وجود هذه الطبقة يقلل تكرار الكود، ويضمن أن الخدمات تتعامل مع نفس الصيغ والأنواع. مثلاً، شكل استجابة البحث معرف في مكان واحد، وهذا يجعل النتائج القادمة من Retrieval Service أو Ranking Service متشابهة وسهلة المعالجة.

## 8. Ranking Engine داخل البنية

يعتبر `RankingEngine` القلب العملي للنظام، لكنه لا يظهر للمستخدم مباشرة. الخدمات تستدعيه لتنفيذ عمليات البحث والترتيب.

يقوم Ranking Engine بالمهام التالية:

- تحميل البيانات المعالجة.
- تحميل نماذج TF-IDF وBM25 من cache.
- تنفيذ البحث بالطرق الفردية.
- تنفيذ طرق الدمج الهجين.
- جلب النص الأصلي للوثيقة من SQLite عند بناء النتيجة النهائية.
- إدارة التحميل الكسول للنماذج الثقيلة مثل Word2Vec وBERT.

بهذا الشكل يبقى المحرك مسؤولاً عن منطق البحث، بينما تبقى الخدمات مسؤولة عن تنظيم الوصول إليه عبر API.

## 9. قاعدة بيانات الوثائق الأصلية

من النقاط المهمة في البنية أن النظام لا يعرض النص المعالج للمستخدم، بل يعرض النص الأصلي المخزن في قاعدة بيانات SQLite:

`data/documents_msmarco.db`

السبب هو أن النص المعالج يكون مناسباً للنماذج لكنه غير مناسب للقراءة البشرية. لذلك تتم عملية البحث باستخدام البيانات المعالجة، وبعد تحديد أفضل الوثائق يتم استخدام `doc_id` لجلب النص الأصلي من قاعدة البيانات.

هذا التصميم يحقق توازناً جيداً:

- سرعة ودقة في البحث باستخدام النصوص المعالجة.
- وضوح في العرض باستخدام النصوص الأصلية.

## 10. ملفات التخزين المؤقت

حتى لا يعيد النظام بناء النماذج عند كل تشغيل، يتم استخدام cache:

`cache/engine_cache_msmarco.joblib`

وملف metadata:

`cache/engine_cache_msmarco_meta.json`

هذه الملفات تسمح للنظام بتحميل النماذج الجاهزة بدلاً من تدريبها من البداية. كما يوجد warmup عند تشغيل Gateway لتحميل المحرك في الخلفية، بحيث لا يدفع أول مستخدم كلفة التحميل الطويلة أثناء البحث.

## 11. مسار الاتصال بين الخدمات

عند إرسال طلب بحث، يكون المسار كالتالي:

1. المستخدم يكتب الاستعلام في واجهة Streamlit.
2. الواجهة ترسل الطلب إلى API Gateway.
3. Gateway يستدعي Query Refinement Service لتحسين الاستعلام.
4. إذا كانت الطريقة فردية مثل TF-IDF، يتم توجيه الطلب إلى Retrieval Service.
5. إذا كانت الطريقة هجينة مثل RRF، يتم توجيه الطلب إلى Ranking Service.
6. الخدمة المناسبة تستدعي Ranking Engine.
7. المحرك يحسب أفضل الوثائق.
8. يتم جلب النص الأصلي من SQLite.
9. تعود النتائج إلى Gateway ثم إلى الواجهة.

هذا المسار يجعل كل جزء مسؤولاً عن خطوة محددة، بدلاً من وضع كل شيء داخل دالة واحدة ضخمة.

## 12. مزايا البنية المستخدمة

توفر البنية الحالية عدة مزايا:

- **وضوح المسؤوليات**: كل خدمة لها دور محدد.
- **سهولة الاختبار**: يمكن اختبار كل خدمة بشكل منفصل.
- **قابلية التوسعة**: يمكن إضافة خدمة جديدة لاحقاً دون إعادة بناء النظام بالكامل.
- **سهولة الصيانة**: تعديل جزء معين لا يتطلب تعديل كل المشروع.
- **تنظيم أفضل للتقرير والعرض**: يمكن شرح النظام كمجموعة مكونات مترابطة.
- **أداء أفضل**: بفضل cache وwarmup والتحميل الكسول للنماذج الثقيلة.

## 13. خلاصة القسم

تم بناء النظام وفق بنية SOA باستخدام FastAPI، مع وجود API Gateway كنقطة دخول موحدة. تم تقسيم المشروع إلى خدمات مستقلة تغطي المعالجة، الفهرسة، الاسترجاع، الترتيب، تحسين الاستعلام، والتقييم. كما تم الاعتماد على SQLite لحفظ النصوص الأصلية، وcache لتسريع تحميل النماذج.

هذه البنية تجعل المشروع أكثر من مجرد تطبيق بحث بسيط؛ فهي تحوله إلى نظام منظم وقابل للتطوير، ويمكن عرضه وشرحه بطريقة واضحة ومقنعة.
